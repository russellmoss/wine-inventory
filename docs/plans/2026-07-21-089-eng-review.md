## 1. Architecture Review

### [P0] (confidence: 10/10) `useVoiceSession.ts:132` — voice-session history diverges from chat history the moment the user types. **COUNCIL MISSED THIS.**

This is the most serious finding in any of the three reviews, and it is *created by the plan's own
new feature*.

`historyRef` is snapshotted once at mount:

```
useVoiceSession.ts:132   const historyRef = useRef<ChatMessage[]>(opts.initialHistory.slice());
useVoiceSession.ts:277   messages: clampHistoryForSend(historyRef.current)   // <- what gets SENT
useVoiceSession.ts:400   historyRef.current.push({ role: "assistant", ... }) // voice turns only
useVoiceSession.ts:469   historyRef.current.push({ role: "user", ... })      // voice turns only
```

Nothing ever syncs a **typed** message into `historyRef`. Today that is harmless: the full-screen
overlay makes typing impossible, so the two histories cannot diverge.

The plan's Requirement "the text composer stays usable during voice" removes exactly that protection.

```
   USER TYPES "log 22.4 Brix for Block 3"
        │
        ├──> items[]           (chat transcript)     ✅ has it
        └──> historyRef.current (what voice SENDS)   ❌ never sees it
                    │
   USER SAYS "actually make it 23"
                    │
                    └──> POST /api/assistant  messages = [ ...no mention of 22.4... ]
                         assistant: "make what 23?"
```

Codex's C3 (serialize the turns) does **not** fix this. Even with perfect serialization, the voice
session is reasoning over a history that is missing everything the user typed. Worse, the failure is
silent and looks like the assistant "forgetting" — which is exactly the class of bug that destroys
trust in an assistant.

**Options:**
- **(a)** Keep the composer usable and sync typed turns into the session. Requires either an exposed
  `appendHistory(turn)` on `VoiceSession`, or re-deriving `initialHistory` and having the panel push
  it down. Touches `useVoiceSession`'s public type — which the plan claims not to change.
- **(b)** Disable the composer while a voice session is live. Contradicts an explicit user decision,
  and removes the "mic misheard the lot number" escape hatch that motivated it.
- **(c)** Keep the composer usable for **text only when voice is `listening`/`idle`** (Codex C3's
  guard) **and** sync the typed turn into `historyRef` via a new `appendHistory` method.

**Recommendation: (c).** It preserves the user's decision, fixes both the concurrency defect and the
divergence defect, and the API addition is one additive method — it does not change voice *behavior*,
which is what the "no engine change" constraint was actually protecting.

**This invalidates the plan's claim that `useVoiceSession` is untouched.** The plan must say so.

### [P1] (confidence: 9/10) Unit 4 silently drops `focusNotice` / `unmatchedBursts`. **COUNCIL MISSED THIS.**

`VoiceOverlay.tsx:180-184` is the **only** render site in the entire codebase:

```
grep focusNotice src/  ->  VoiceOverlay.tsx:180,181,182  +  src/lib/voice/focus.ts (producer)
```

Unit 4's keep-list names the error line, the focus control, Interrupt, and the proposal card.
`focusNotice` is in neither the keep-list nor the delete-list. It just evaporates.

That notice is the entire user-facing surface of voiceprint focus — the "I didn't catch that, it
didn't sound like you" feedback, which escalates to `--danger` at `unmatchedBursts >= 2`. Delete the
render site and `src/lib/voice/focus.ts` becomes live logic computing a string nobody ever sees, still
covered by `test/voice-focus.test.ts` — green tests over a dead feature.

**Fix (no real alternatives — applying directly):** add `focusNotice` to Unit 4's explicit keep-list,
rendered in the control row with the same `unmatchedBursts >= 2` danger escalation. In a 440px panel
it must wrap, not use the overlay's `maxWidth: 460` centered treatment.

### [P2] (confidence: 8/10) Unit 6 is under-specified and would not compile as described.

Unit 5's composer changes, vertical-space reclaim, and panel mount point are all gated on
`embedded &&`. Unit 6 says "use `VoiceInlinePanel` on the full-page `/assistant` route too" — but that
route is `embedded === false`, so none of Unit 5's layout work applies there. Unit 6 as written
inherits a panel with no defined mount point, no pinned-card placement, and no orb host.

**Fix:** Unit 6 must explicitly state the non-embedded layout: orb beside the `<h1>` at
`AssistantChat.tsx:740`, panel mounted above the composer in the non-embedded branch, and the
vertical-space reclaim skipped (the full page has room). Or drop Unit 6. See D2.

### [P3] (confidence: 7/10) No stated interaction with the repo's verify gates.

`src/lib/voice/inline-ui.ts` is a new pure module. Two repo gates could plausibly fire and `/work`
should not have to guess:
- `verify:ai-native` guards `*-core.ts` → tool wiring. `inline-ui.ts` is **not** a `-core.ts`, so it
  does not trip. (Same reasoning as the `cost-display.ts` precedent.)
- `.claude/hooks/inject-brain-context.mjs` governs `src/lib/{ledger,tenant,cost,compliance,transform,accounting,commerce,auth}`. `src/lib/voice/` is **not** governed → no invariant note, no `INVARIANTS.md` entry.

**Fix (applying directly):** state both in the plan so nobody re-derives it.

## 2. Code Quality Review

### [P1] (confidence: 9/10) E3 — Unit 1's "controls model" duplicates `src/lib/voice/focus.ts`.

`focus.ts` already owns focus-mode state and already exposes a label (`test/voice-focus.test.ts:31`,
*"labels modes for the overlay"*). Unit 1 proposes a `voiceControlsModel` that re-derives "which focus
action to offer" from `focusMode` + `profileState` — the same domain, in a second module.

Given the stated preference (*"DRY is important — flag repetition aggressively"*), splitting focus
logic across two files in `src/lib/voice/` is the wrong shape.

**Fix (applying directly):** Unit 1 keeps `voiceStatusLabel` and `voiceAnnouncement` (genuinely new,
presentation-only). The focus-action derivation moves into `focus.ts` beside the existing label logic,
and `test/voice-focus.test.ts` gets the new cases. `inline-ui.ts` stays small and single-purpose.

### [P3] (confidence: 8/10) Stale reference: `test/voice-focus.test.ts:31` says *"labels modes for the overlay."*

Unit 6 deletes the overlay. Per the diagram-maintenance rule, rename in the same commit that removes
the thing it names.

### No other DRY or organization issues found.
The `ProposalCard` extraction to module scope is the right call — it is currently a private component
inside a file being deleted, and two call sites will need it.

## 4. Performance Review

**No issues found.** Concretely:
- Orb goes 220px → 28px, one instance, same single rAF loop. Strictly less compositor work.
- The council's C1 fix (publish status only on the 6-value enum, `getLevel` behind a ref) already
  removes the only re-render hazard. Without it, a 60fps status object would re-render the dock and
  drag the whole 1388-line `AssistantChat` subtree with it — that would have been the real perf bug,
  and it is already caught.
- No queries, no data layer, no network change. `AssistantChat` is lazy; `VoiceInlinePanel` should
  stay lazy the same way the overlay is (`AssistantChat.tsx:25-26`) — noting so it is not lost in
  the swap.
- Dock drag writes styles imperatively on pointermove while the orb's rAF runs. At 28px this is
  noise, not a problem.
