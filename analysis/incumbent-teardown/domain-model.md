## 1. vintrace's implicit data model

vintrace is a **batch-centric, mutable-record** system with a strong *inventory/costing* backbone.
The central noun is the **Wine Batch** (aka "batch code", "bulk product"). The implicit model:

- **Wine Batch** — the durable production identity. It is a *set of vessels* holding a bulk product
  (`vintrace: vintrace-web/winemaking/using-the-batch-explorer.md` — the Batch Explorer lists "current
  and depleted bulk wines by batch"; a batch has an **In Production** tab listing its vessels, plus
  **Dispatched / Bottled / Costs** tabs). A batch carries properties: batch code, **vintage**,
  **variety/designated variety**, **program**, **owner**, **bond** (derived), composition, tax class.
- **Batch code** is a **mutable label that rewrites history**: "Changing the batch code updates **all
  historical references** to that code" (`vintrace: vintrace-web/winemaking/changing-a-wine-batch-s-properties.md`).
  Properties like Program are editable behind an *Advanced data management* permission with a warning.
- **Batch identity is generated by "Auto-Codes"** — a fully **configurable, winery-defined template
  engine** (`vintrace: setup-and-admin/configuration/configuring-and-using-auto-codes.md`). Policies
  exist for ~17 record types (batch, batch number, barrel, barrel group, bin, crush load, fruit intake,
  trial blend, tirage group, sample set, stock item batch, …). Each policy is an ordered list of *code
  elements* (Attribute / Inc / Text), e.g. `Vintage + Variety + Region + 3-digit Inc`. There is a
  **"Fraction Type" attribute** baked into batch codes (M/MU Must, F/FR Free Run, P/PR Pressings,
  S/SN Saignée, **L/LS Lees**, D/DR Drainings, C/CO Combined…) — vintrace encodes press/lees fractions
  *into the name itself*. Custom codes add a per-winery text token (e.g. `JE`). This is a first-class,
  admin-configurable naming layer — vintrace's strongest identity feature.
- **Vessel** — tanks, barrels, bins, kegs, gyro cages. Barrels can be grouped into a **Barrel Group**
  (a managed collective vessel) and individually extracted via a **Break Barrel** operation, which can
  *reassign the batch and/or barrel-group name* of the broken-out barrels
  (`vintrace: vintrace-web/barrel-management/breaking-barrel-out-of-a-barrel-group.md`). Vessels have
  their own auto-code policies and an *overfill-prevention* setting.
- **Operation / Job / Work Order** — cellar work is a **Job** managed on the **Job Management console**
  with statuses **draft → ready → in progress → submitted → completed**
  (`vintrace: vintrace-web/work-orders/job-management-console.md`). Operations (Transfer/Rack/Blend,
  Multi-Transfer many↔one, Change Batch, Change Ownership, Break Barrel, Press, Topping, Additions,
  Measurement, Bulk Dispatch, Bottling…) mutate batch state.
- **Owner / Bond** — a batch has an owner (Address Book contact); **bond is derived from location**
  *unless* an AP/composite bond attaches to the owner, in which case owner-bond wins
  (`vintrace: vintrace-web/compliance/transferring-wines-between-bonds-us.md`).
- **Additive lot / stock** — additives carry batch/lot tracking + expiry with an Inventory module;
  a Wine Addition Impact Report traces which additive lot touched which wine batch
  (`vintrace: vintrace-web/winemaking/lot-tracking-traceability.md`).

**Correction model = mutate-in-place + rollback/replay.** vintrace lets you **Correct** a bulk intake's
composition directly (`vintrace: vintrace-web/winemaking/fixing-a-wine-s-composition.md` — "vintrace
updates any blends that have occurred using the wine", gated by Advanced Data Management + a
*Reason for Correction*). Where in-place correction is impossible (volume errors), you use
**Rollback** or **Rollback & Replay**: rollback reverses the selected op *and every subsequent op that
touched the wine*, restoring prior state as "if the selected operation never happened"; replay parks the
subsequent ops in a special **Replay work order** to re-save
(`vintrace: faq/common-questions/how-do-i-use-rollback-and-replay-to-fix-data-entry-errors.md`).
Crucially: "**Operations that change the volume cannot be reversed**… toppings, transfers, treatments,
changes in ownership, changes in batch, press cycles, extractions, and all sparkling operations" — those
force a rollback, and large rollbacks may require **contacting vintrace support**.

### vintrace awkwardness (flagged)
- **Change Ownership needs a follow-up zero-volume Measurement to "lock" the bond change** — the exact
  user-flagged quirk, **confirmed**: after a Change Ownership op "you'll need to perform a **Measurement
  operation on the wine without changing the volume** (zero loss/gain)… The Measurement operation is
  important for reporting purposes as it **locks in the bond change** as of the date/time"
  (`vintrace: vintrace-web/compliance/transferring-wines-between-bonds-us.md`). Three different ops
  (Change Batch / Transfer / Change Ownership) all mutate bond, each with different reporting side
  effects and a warning dialog — bond change is a fragile, multi-path, follow-up-required flow.
- **Volume-changing ops are irreversible without a chain rollback** — no per-op inverse; you unwind the
  whole downstream chain and rebuild it via a Replay work order (or call support). Correction is a
  destructive rewrite of history, not an appended event.
- **Batch-code edit silently rewrites all history** — no separate display-name; identity and label are
  the same string, and editing it retroactively restamps every historical reference.
- **Fraction type / lees encoded into the code string** — semantics (Saignée, Lees, Pressings-heavy)
  live in the *name*, not as a typed attribute, so downstream logic must parse the code.

## 2. InnoVint's implicit data model

InnoVint is a **lot-centric, action-log** system, cloud/mobile-native. The central noun is the **Lot**.

- **Lot** — the durable production identity, of a **type**: **Fruit Lot**, **Juice/Wine Lot**, or
  **Case Goods Lot**. A Juice/Wine lot's attributes: Bond, **Lot Code**, **Lot Name**, **Lot Color**
  (Red/White/Rosé/Orange), **Lot Style** (Still/Sparkling), Expected Yield, **Tax Class**, **Stage**,
  **Tags**, **Owner(s)** (`innovint: make/lots/juice-wine-lot-attributes.md`). The **Lot Details Page**
  is "home base" with Dashboard + tabs: Analysis, Additives, Vessels, History, Timeline, **Composition**
  (vintage/varietal/appellation/vineyard roll-ups), Cost (`innovint: make/lots/lot-details-page.md`).
- **Fruit Lot** — required to receive fruit; **always 100% single vintage/vineyard/block**; code
  auto-prefixed `FL-` (e.g. `FL-18ZN-PTV-A`), editable once vineyard+block are chosen; carries an
  Expected Yield (default 150 gal/ton) that flows into the derived juice/wine lot
  (`innovint: harvest/harvest-workflow-fermentation-tracking/how-to-create-a-fruit-lot.md`). **Once
  analysis/activity is recorded, vintage/vineyard/block are frozen** — "these compositional pieces will
  follow your wines from the fruit lot through to bottling"; a mistake means *delete the lot and start
  over*.
- **Lot Code + Lot Name are freely mutable and rewrite history** — "You can change the lot code, lot
  name, lot color or lot style **at any point in time**… These lot properties will change to display
  the new code/name **throughout the entire history** of the lot"
  (`innovint: make/lots/changing-lot-properties.md`). Code changes are **not tracked as actions** (they
  recommend manually adding a note), but are captured in a separate **Lot Properties History** report.
  Codes accept alphanumerics + `-` + `_`, no accents. InnoVint auto-generates a code you may overwrite;
  there is **no configurable template engine** like vintrace Auto-Codes.
- **Vessel** — tanks/barrels/kegs. The **Vessel Details Page** shows the *current* lot, fill, stage,
  location + history; a vessel can only be *deleted* if it never held a lot, else *archived*
  (`innovint: make/vessels/vessel-details-page.md`). Vessel occupancy is implicit: a lot "fills" one or
  more vessels; the vessel knows its current lot.
- **Action** — the atomic recorded event: **Receive Fruit, Process to Tons/Volume, Weight Transfer,
  Rack, Transfer, Rack & Return, Blend, Barrel Down, Top Off, Filter, Addition, Analysis, Volume
  Adjustment, Tax Class change, Custom action, Bottle, B2B (bond-to-bond), Remove Taxpaid**. Actions are
  either **direct** (recorded immediately) or **generated by a submitted Work Order task**.
- **Work Order** — a *plan/request* wrapping one or more **Tasks**; statuses **Not started → Started →
  Completed → Submitted** (`innovint: make/work-orders/using-work-orders-in-innovint.md`). The written
  task is "the request *for* an action"; **submitting the task records the underlying action** and
  writes inventory. Tasks can be skipped; work orders are deletable only before submission; recurring
  WOs and templates exist. Three submission-time semantics: as-of-now / as-of-specific-datetime /
  as-of-completion. Individual task submit cascades volumes to dependent tasks.
- **Lees Lot** — a **first-class sub-lot**. Rack / Transfer / Blend / Filter / Barrel Down / Top Off /
  Rack & Return all offer "**save or discard the lees**" into a new or existing lees lot
  (`innovint: make/movement-actions/how-to-record-a-rack.md`;
  `innovint: make/movement-actions/how-to-record-a-blend.md`;
  `innovint: make/recording-actions/creating-a-new-lot-within-an-action-or-task.md`; feature shipped
  `innovint: product-updates/product-updates-2017-2022/11-16-20-release-notes-…-lees-lots.md`). Saved
  lees inherit the parent's varietal composition and are proportionally subtracted from the blend volume.
- **Bond / Tax Class / Owner** — Bond is a lot attribute; **Tax Class is a dated, editable declaration**
  that "maps directly to the TTB Report" and can be set at creation / on the lot details page (effective
  any date) / via the Tax Class Declaration report (`innovint: make/compliance/declare-or-edit-tax-class.md`).
  Owner is a permissions/custom-crush tag on the lot.

**Correction model = edit/delete-in-place, chained by "dependent actions".** "Once you've clicked
Submit… you cannot go back in time and undo that submission"; you must **edit** or **delete-and-redo**
(`innovint: make/recording-actions/how-to-edit-or-delete-recorded-actions.md`). Only the **most recent
action with no dependents** is freely deletable; **movement actions on the involved vessels chain as
"dependent actions"** and must be deleted (then re-recorded) first. Admins can edit an action *plus up
to 50 dependent actions*; beyond that it errors. Deleted/edited actions get a "deleted"/"edited" tag and
are exportable, but the mutation is **in place**, not an appended compensating event. Tax-class changes
and analyses/additions are explicitly *not* dependent actions and can be freely edited/deleted.

### InnoVint awkwardness (flagged)
- **No native "split a lot" operation** — splitting requires a **two-transfer phantom-vessel
  workaround**: transfer the portion to an empty "phantom" vessel under a new lot code, then transfer it
  back to its real vessel, with notes explaining no physical movement occurred
  (`innovint: guidance-faqs/frequently-asked-questions/how-to-split-a-lot.md`). Barrel splits need pre-
  tagging the barrels.
- **No native "Blend & Return"** — a documented **3-step workaround** (tag vessels → Blend → Barrel
  Down), with dependency ordering and (1 of 2)/(2 of 2) WO-title conventions
  (`innovint: guidance-faqs/frequently-asked-questions/how-do-i-record-a-blend-return.md`).
- **Rack & Return can't top off** — the action refuses any net *gain* on a vessel
  ("Positive volume change in a drain for vessel…"), forcing a separate Top Off action
  (`innovint: make/movement-actions/how-to-record-a-rack-and-return.md`).
- **Correction = delete the chain and re-enter** — no per-op inverse; deleting a mid-history action
  demands deleting every downstream dependent action first, even on *unrelated lots* that later used the
  same vessel (`…how-to-edit-or-delete-recorded-actions.md` FAQ). The 50-dependent-action edit cap and
  "contact support" escape hatch mirror vintrace's rollback pain.
- **Lot code rewrite is untracked as an action** — history silently restamps; they *recommend a manual
  note*. Fruit-lot composition mistakes are unfixable ("delete and start over").
- **Blend into an occupied lot mis-maps the TTB report** — blending across tax classes can wrongly
  populate 5120.17 Lines 5/20 (`innovint: make/movement-actions/how-to-record-a-blend.md` FAQ;
  `innovint: guidance-faqs/frequently-asked-questions/blending-across-tax-classes.md`).

## 5. Convergence / divergence / both-fail map

**CONVERGE (table stakes — must match):**
- Lot/batch as the central production identity spanning multiple vessels; occupancy tracked per vessel.
- Work-order → task → recorded-operation pipeline with templates, assignment, statuses, mobile
  completion, skip. (Both have mature WO engines; ours matches structurally.)
- Dated tax-class / bond as a lot attribute feeding the TTB 5120.17.
- Composition/varietal/appellation roll-up over lineage; additive-lot traceability.
- Barrel-level operations and grouping are heavily documented in both (194/172 `barrels`-tagged
  articles) — cellar crews expect first-class barrel handling.

**DIVERGE (design decision — where we can win):**
- **Correction philosophy.** vintrace = in-place Correct + destructive chain rollback (volume ops
  irreversible, support-assisted); InnoVint = edit/delete-in-place gated by dependent chains (≤50 cap).
  Cellarhand = **append-only compensating events, never mutate/delete**. This is the single biggest
  architectural divergence and our clearest moat.
- **Identity vs label.** Both incumbents **fuse code and identity and rewrite all history on rename**.
  Cellarhand splits surrogate-id from a label — *but* pins the label immutable, so we currently offer
  *worse* rename UX than either. The right answer is our id/label split **plus** a mutable display layer.
- **Naming template.** vintrace has a rich configurable Auto-Code engine (win for them); InnoVint has
  none; Cellarhand hardcodes a scheme (trailing both).
- **Split & lees.** Cellarhand has **native split + lineage** (both incumbents fake split via phantom
  vessels / Change Batch). InnoVint has **first-class lees lots** (Cellarhand + vintrace lack them).

**BOTH FAIL (differentiation opportunity):**
- **Ownership / bond change is a ritual, not an event.** vintrace's Change Ownership *requires a
  follow-up zero-volume Measurement to lock the bond*; three overlapping ops (Change Batch / Transfer /
  Change Ownership) each mutate bond differently with warning dialogs. InnoVint treats owner as a mutable
  tag with no locking event. Neither models a bond as a first-class instrument with clean transfer-in-
  bond events. **An append-only "change ownership / transfer-in-bond" op with no follow-up step is a
  wedge.**
- **Corrections require deleting/rebuilding downstream chains** in both — brittle, cap-limited, support-
  ticket-prone. Our LIFO compensating-event unwind is strictly better if we make it self-serve and
  legible.
- **Renaming quietly rewrites audit history** in both (InnoVint doesn't even log the code change as an
  action). A tamper-evident, fully-audited rename is an easy trust win.
- **Fruit-lot composition is frozen / "delete and start over"** (InnoVint) — an immutable-ledger system
  can offer a clean corrective re-composition without data loss.

## 6. Recommendations

1. **Lead every migration pitch with the correction model.** Both incumbents destroy or rewrite history
   to fix errors (vintrace rollback/replay + irreversible volume ops; InnoVint delete-the-dependent-
   chain, 50-action cap, "contact support"). Cellarhand's append-only CORRECTION with LIFO unwind is a
   genuine architectural moat — make it a first-run demo, not a footnote.
2. **Split identity from label the rest of the way.** Keep the immutable surrogate `id` + immutable
   `code` for the ledger, but add a **mutable, fully-audited `displayName`** so wineries get the
   rename UX both incumbents offer *without* rewriting history. This closes our one UX regression versus
   both competitors (§5 divergence) while preserving the invariant.
3. **Build a configurable naming-template engine.** vintrace's Auto-Codes (ordered elements, per-record-
   type policies, custom tokens) is its strongest identity feature and a real switching-cost anchor for
   their customers. Our hardcoded scheme trails both; a per-tenant template (reusing the pure
   `buildLotCode` seam) is high-leverage for Phase-13 migration parity.
4. **Add two missing cellar primitives that both crews expect: lees sub-lots and barrel groups.** Lees
   lots are first-class in InnoVint across six action types and inherit composition — table stakes for
   red-wine and barrel programs. Barrel groups (+ break-barrel) are pervasive in both corpora. Both fit
   our ledger cleanly (a lees lot = a child lot via a lineage edge on a rack; a barrel group = a vessel
   collective).
5. **Model ownership + bond as appended ledger ops, killing the follow-up ritual.** Directly target
   vintrace's worst-documented quirk: a single `CHANGE_OWNERSHIP` / `TRANSFER_IN_BOND` op that stamps the
   bond change atomically (no separate zero-volume Measurement), drives both 5120.17 sides, and triggers
   the cost recompute custom-crush needs. Requires a first-class bond entity (currently `[ABSENT]`).
6. **Make dependent-action corrections a non-issue.** Our LEDGER-11 already blocks unsafe corrections and
   supports LIFO chain-unwind; expose it as a legible, self-serve "undo this and everything after it"
   flow so wineries never hit the 50-action cap / support-ticket wall that both incumbents impose.
7. **Write the missing lifecycle writers.** `Lot.status` DEPLETED/ARCHIVED are declared but never
   written (§8 of the state brief), and `LotLineage.TRANSFORM` is unused. Both incumbents lean on
   archive semantics (archive-not-delete once activity exists); we should implement a real close/archive
   lifecycle to match their inventory hygiene.
```
